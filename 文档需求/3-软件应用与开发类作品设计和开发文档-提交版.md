## 第一章 需求分析

### 1.1 开发背景

本作品面向中小学科学课程中的“种植观察、环境监测、实验报告与过程评价”场景。传统教学中普遍存在以下问题：

1. 传感器数据采集分散，缺少统一可视化与可追溯记录。
2. 教师布置实验任务、收集报告、批改反馈流程割裂，效率低。
3. 学生对植物生长与环境变量关系的理解依赖单次观察，难以形成数据驱动结论。
4. 缺少适配课堂的角色权限系统，难以在同一平台支持学生、教师、管理员协同。

因此，本项目建设“智慧大棚 IoT 教学系统”，将设备监测、教学内容、实验报告、植物档案、小组协作和 AI 科学助手整合在同一 Web 平台，实现课堂可用、可演示、可评估的完整闭环。

### 1.2 目标用户与典型场景

| 用户角色 | 典型诉求 | 典型场景 |
|---|---|---|
| 学生 | 查看实时数据、完成学习与实验任务、提交报告 | 科学课实验观察、课后复盘 |
| 教师 | 布置任务、批改评分、管理教学内容、组织小组 | 课堂教学与作业管理 |
| 管理员 | 管理用户班级设备、运营系统、导出日志 | 校内平台运维与治理 |
| 公开访客 | 无登录查看大屏实时状态 | 公开课展示、成果汇报 |

### 1.3 竞品与对标分析（按类型）

| 对标类型 | 优势 | 不足 | 本作品改进点 |
|---|---|---|---|
| 通用在线教学平台 | 课程与作业流程成熟 | IoT 监测能力弱，缺少设备控制与实时数据联动 | 增加设备遥测、控制、数据导出与大屏展示 |
| 通用 IoT 监控平台 | 实时监控与设备接入能力强 | 教学内容、作业批改、学习闭环不足 | 增加教学内容、实验任务、批改与植物档案 |
| 单功能校园小程序 | 上手快、轻量 | 模块单一，扩展性与权限治理弱 | 提供多角色 RBAC、模块化架构与可扩展 API |

### 1.4 功能与性能目标

功能目标：

1. 提供学生/教师/管理员三级角色权限控制。
2. 支持设备实时监测、历史查询、远程控制和数据导出。
3. 支持教学内容管理、学习记录、评论互动、通知提醒。
4. 支持实验任务布置、学生提交、教师批改、AI 辅助点评。
5. 支持植物档案和生长记录时间轴管理。
6. 支持小组合作与角色分工。
7. 支持 AI 科学助手（含会话管理、RAG 检索、联网来源引用）。

性能与质量目标：

1. 监测数据支持 5 秒轮询刷新与 WebSocket 实时推送。
2. 关键分页接口具备边界参数校验（页码/页大小非法输入可正确拒绝）。
3. 登录接口支持失败频控与重试保护。
4. 系统支持 Docker 与本地双模式部署，便于教学演示。

### 1.5 AI 能力需求（项目功能侧）

项目面向课堂问答场景引入 AI 能力，重点需求包括：

1. 科学问题解释可结合本地教学内容（RAG）与联网检索。
2. 回答需支持来源引用对齐，保证可追溯。
3. 深度思考模式可切换，满足不同教学深度。
4. 会话可管理（新建、切换、重命名、删除、置顶）。

## 第二章 概要设计

### 2.1 总体技术架构

系统采用前后端分离 + IoT 数据接入架构：

1. 硬件端：ESP32 上报传感器数据。
2. 后端：FastAPI 提供 REST、WebSocket、SSE 与业务服务。
3. 前端：Vue 3 + TypeScript 构建单页应用。
4. 数据层：MySQL 存储业务数据，Alembic 管理迁移。
5. AI 层：LangChain 编排 DeepSeek + Chroma 向量检索。

架构链路：

ESP32 -> FastAPI -> MySQL  
Vue 前端 <-> FastAPI（HTTP/WebSocket/SSE）

### 2.2 功能模块划分

| 模块 | 子模块 | 主要功能 |
|---|---|---|
| 认证与权限 | 登录、注册、注销、RBAC | JWT 鉴权、角色守卫、频控 |
| 设备与监控 | 设备管理、遥测、控制、历史 | 实时监测、执行器控制、导出 CSV/Excel |
| 教学内容 | 内容 CRUD、学习记录、评论互动 | 标签组织、学习进度、统计概览 |
| 实验报告 | 任务发布、提交、批改 | 文件上传下载、评分与反馈 |
| 植物档案 | 档案管理、生长记录 | 阶段标记、时间轴展示 |
| 小组协作 | 小组与成员管理 | 分工协作、跨班迁移（管理员） |
| 线下商城 | 商品管理与展示 | 发布、搜索、图片上传 |
| 个人中心 | 资料、头像、通知 | 未读数、待办统计 |
| AI 助手 | 会话管理、问答、流式输出 | RAG、联网检索、引用对齐 |
| 运维治理 | 日志与导出、迁移维护 | 操作审计、版本迁移、脚本回归 |

### 2.3 层次结构与调用关系

后端分层：Route -> Dependency -> Service -> Core/DB。  
前端分层：View -> Component/Composable -> API -> Backend。

调用特征：

1. 路由层仅做参数校验、权限检查和响应组织。
2. 业务逻辑集中在 services 层，便于复用和测试。
3. 核心安全策略（JWT、权限判定、校验）集中在 core 层。
4. 前端通过 API 层统一封装请求与错误处理。

### 2.4 关键接口与数据流

1. 遥测流：设备上报 `/api/telemetry` -> 写入数据库 -> 广播 WebSocket。
2. 控制流：教师/管理员调用 `/api/control/{device_id}` -> 更新设备执行状态。
3. 作业流：教师创建任务 -> 学生提交 -> 教师批改 -> 学生查看反馈。
4. AI 流：用户提问 -> RAG/联网检索 -> 模型生成 -> 引用对齐 -> 返回答案与来源。

### 2.5 安全与权限概要

1. JWT + Token 黑名单实现认证与注销。
2. 三级角色：学生、教师、管理员。
3. 班级-设备绑定控制可见范围。
4. 教师资源遵循“全校可见、仅本人可写”；管理员全权限。
5. 登录失败频控：同一窗口内失败超阈值触发限制。

## 第三章 详细设计

### 3.1 界面设计

系统界面采用“角色首页 + 业务模块页 + 公共组件”方式组织：

1. 首页与导航：按角色重定向到 StudentHome/TeacherHome/AdminHome。
2. 监控页：展示温湿度、土壤、光照等图表与设备状态面板。
3. 教学内容页：支持筛选检索、内容详情、学习进度与评论互动。
4. 实验报告页：支持任务列表、提交、批改及状态追踪。
5. 植物档案页：支持档案列表、生长记录和阶段管理。
6. 小组页：支持组内成员与角色协作。
7. AI 助手：浮窗式交互，支持 Markdown、公式、图示渲染。

### 3.2 数据库设计

系统包含 20 个核心数据表，按域分组如下：

1. 用户与组织域：users、classes、class_device_binds。
2. 设备遥测域：devices、sensor_readings。
3. 教学内容域：teaching_contents、student_learning_records、content_comments、content_comment_likes。
4. 作业实验域：assignments、assignment_submissions。
5. 植物与小组域：plant_profiles、growth_records、study_groups、group_members。
6. AI 与日志域：ai_conversations、ai_conversation_messages、user_operation_logs、user_notifications。
7. 业务扩展域：market_products。

关键关系：

1. 用户与班级、班级与设备通过绑定关系控制可见范围。
2. 作业与提交通过 assignment_id 关联，支持批改者与提交者分离。
3. AI 会话与消息一对多关联，支持会话级历史追踪。

### 3.3 关键业务流程设计

流程 A：实验闭环流程  
教师创建任务 -> 学生提交报告（可带附件） -> 教师评分与反馈 -> 学生查看 graded 状态。

流程 B：遥测监控流程  
ESP32 上报 -> 后端入库 -> 前端轮询/推送更新 -> 用户查看趋势并导出数据。

流程 C：AI 问答流程  
用户输入问题 -> 判断是否大棚场景 -> 进行 RAG 与联网检索 -> 模型生成回答 -> 引用编号与来源对齐 -> 审计记录。

### 3.4 关键技术与创新点

1. 教学与 IoT 融合：将实时设备数据与实验任务、植物观察直接关联。
2. 角色化架构：同一系统支持学生/教师/管理员差异化能力，且权限边界明确。
3. AI 可追溯：回答中引用编号与来源一一对齐，降低“黑盒回答”风险。
4. 可维护性工程：后端 Route-Service 分层 + 前端 Composable 抽离，提高迭代效率。
5. 双模式部署：本地开发与 Docker 部署兼容，适配课堂演示。

### 3.5 AI 功能设计说明（项目功能侧）

1. 模型编排：支持基础问答模型与深度思考模型切换。
2. 检索增强：支持本地知识库向量检索，减少脱离课程上下文的回答。
3. 智能联网：支持天气/通用检索并给出来源链接。
4. 输出形态：支持流式输出与富文本渲染，便于课堂讲解。
5. 审计能力：记录来源、耗时、Token 估算和回退原因。

## 第五章 安装及使用

### 5.1 环境要求

1. Python 3.10+
2. Node.js 18+
3. MySQL 8.0+

### 5.2 安装与初始化

1. 安装后端依赖：`pip install -r requirements.txt`
2. 初始化数据库：`python init_db.py`
3. 执行迁移：`python -m alembic upgrade head`

### 5.3 启动步骤

1. 启动后端：`python -m uvicorn main:app --host 0.0.0.0 --port 8000`
2. 启动前端：在 frontend 目录执行 `npm install` 和 `npm run dev`
3. 可选启动模拟器：`python simulate_esp32.py`

### 5.4 典型使用流程

教师流程：

1. 登录教师账号。
2. 创建教学内容与实验任务。
3. 绑定班级设备并查看监控数据。
4. 批改学生提交并给出反馈。

学生流程：

1. 登录学生账号。
2. 查看学习内容与实验任务。
3. 提交实验报告（可附文件）。
4. 查看批改结果与建议。

公开展示流程：

1. 打开大屏页面。
2. 展示实时数据、设备状态与趋势信息。

### 5.5 维护与升级

1. 版本升级后执行 `python -m alembic upgrade head`。
2. 改动后执行最小回归：前端 build、后端 compileall、稳定性探测脚本。
3. AI 相关改动后增加 AI 回归探测。

## 第六章 项目总结

### 6.1 项目成果

本项目完成了“教学管理 + IoT 监测 + AI 辅助 + 数据治理”的一体化 Web 应用设计与开发，具备完整课堂应用场景与评审可验证链路。

### 6.2 主要难点与解决思路

1. 多角色权限边界复杂：通过 RBAC 与资源归属策略分层解决。
2. 教学与设备数据融合难：通过班级-设备绑定与统一数据模型解决。
3. AI 结果可追溯难：通过引用对齐与审计记录机制解决。
4. 演示稳定性要求高：通过脚本化回归与边界态修复提升稳定性。

### 6.3 团队能力提升

在本项目中，团队在以下方面得到提升：

1. 全栈协同开发与模块化架构设计能力。
2. 面向课堂场景的产品需求分析与交互设计能力。
3. 自动化测试与回归治理能力。
4. AI 能力工程化落地与风险控制意识。

### 6.4 后续升级方向

1. 增加课堂级统计看板与学习画像分析。
2. 增强移动端适配与离线数据缓存能力。
3. 持续完善 AI 评估机制与高质量课程语料库。
4. 引入更细粒度的运维监控与告警能力。

### 6.5 推广价值

本项目可作为中小学科学课数字化实践平台，具备较强复用与推广潜力：

1. 可用于校内实验教学标准化。
2. 可支撑公开课、社团活动和竞赛训练。
3. 可作为“教学数据化 + 智能化”融合应用示范样板。
